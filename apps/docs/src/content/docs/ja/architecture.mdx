Daytona は AI エージェント向けに、**完全なコンポーザブルコンピュータ** — [サンドボックス](/docs/ja/sandboxes) — を提供します。Daytona プラットフォームは、それぞれが特定の役割を持つ複数のプレーンコンポーネントで構成されています。

* [インターフェースプレーン](#interface-plane) は Daytona と対話するためのクライアントインターフェースを提供します。
* [コントロールプレーン](#control-plane) はすべてのサンドボックス操作をオーケストレーションします。
* [コンピュートプレーン](#compute-plane) はサンドボックスインスタンスを実行および管理します。

### インターフェースプレーン \{#interface-plane\}

インターフェースプレーンは、ユーザーとエージェントが Daytona と対話するためのクライアントインターフェースを提供します。次のコンポーネントはインターフェースプレーンの一部であり、すべてのユーザーとエージェントが利用できます。

* **SDK**: プログラムからサンドボックスを管理するための [Python](/docs/ja/python-sdk)、[TypeScript](/docs/ja/typescript-sdk)、[Ruby](/docs/ja/ruby-sdk)、および [Go](/docs/ja/go-sdk)
* [CLI](/docs/ja/tools/cli): サンドボックスを直接操作するためのコマンドラインインターフェース
* [Dashboard](https://app.daytona.io/dashboard/): サンドボックスの管理とモニタリングのためのウェブインターフェース
* [MCP](/docs/ja/mcp): AI ツール統合のための Model Context Protocol サーバー
* [SSH](/docs/ja/ssh-access): 実行中のサンドボックスへのセキュアシェルアクセス

### コントロールプレーン \{#control-plane\}

コントロールプレーンは、Daytona プラットフォームにおける中核的な制御レイヤーです。すべてのクライアントリクエストを受け取り、サンドボックスのライフサイクル全体を管理し、サンドボックスをランナーにスケジューリングし、インフラストラクチャ全体の状態の整合性を継続的に維持します。コントロールプレーンには次のコンポーネントが含まれます:

* [API](#api) は認証、サンドボックスのライフサイクル管理、およびリソース割り当てを処理します。
* [Proxy](#proxy) は外部トラフィックをサンドボックスにルーティングし、サービスへの直接アクセスを可能にします。
* [Snapshot builder](#snapshot-builder) はサンドボックスの [スナップショット](/docs/ja/snapshots) を構築および管理します。
* [Sandbox manager](#sandbox-manager) はサンドボックスのライフサイクル管理と状態の整合性維持を行います。

#### API \{#api\}

API は NestJS ベースの RESTful サービスであり、すべてのプラットフォーム操作の主なエントリポイントとして機能し、認証、サンドボックスのライフサイクル、スナップショット、ボリューム、およびリソース割り当てを管理します。[snapshot builder](#snapshot-builder) と [sandbox manager](#sandbox-manager) は、API 内部で動作するプロセスとして実行されます。API は次の内部サービスおよびコンポーネントと統合されています。

* **Redis** はキャッシュ、セッション管理、および分散ロックを提供します
* **PostgreSQL** はメタデータと設定のための主要な永続ストアとして機能します
* **Auth0/OIDC provider** は OpenID Connect を介してユーザーおよびサービスを認証します。API は組織レベルのマルチテナンシーを実現しており、各サンドボックス、スナップショット、およびボリュームは特定の組織に属し、アクセス制御は組織の境界で適用されます
* **SMTP server** は、組織への招待、アカウント通知、およびアラートメッセージのメール配信を処理します
* [Sandbox manager](#sandbox-manager) はサンドボックスをランナー上にスケジューリングし、状態を同期し、サンドボックスのライフサイクル管理ポリシーを適用します
* **PostHog** は、モニタリングと改善のためにプラットフォームの分析情報と使用状況メトリクスを収集します

API からサンドボックスを操作する方法については、[API](/docs/ja/tools/api) および [Toolbox API](/docs/ja/tools/api#daytona-toolbox) のリファレンスを参照してください。

#### プロキシ \{#proxy\}

プロキシは、ホストベースのルーティングを用いて外部トラフィックを正しいサンドボックスへルーティングする専用の HTTP プロキシです。各サンドボックスには `{port}-{sandboxId}.{proxy-domain}` でアクセスでき、ここで port はサンドボックス内で動作しているサービスに対応します。プロキシは、指定されたサンドボックスに対するターゲットのランナーを特定し、認証ヘッダーを付与してリクエストを転送します。HTTP および WebSocket の両プロトコルをサポートします。

#### Snapshot builder \{#snapshot-builder\}

Snapshot builder は API プロセスの一部であり、Dockerfile または [コンテナレジストリ](#container-registry) 上のあらかじめビルドされたイメージから、サンドボックスの[スナップショット](/docs/ja/snapshots) を作成する処理全体をオーケストレーションします。ランナーと連携してイメージのビルドまたは取得を行い、その後、それらのイメージを OCI Distribution Specification を実装した内部スナップショットレジストリにプッシュします。

#### サンドボックスマネージャー \{#sandbox-manager\}

サンドボックスマネージャーは API プロセスの一部として動作し、サンドボックスをランナーにスケジュールし、状態を同期し、[サンドボックスライフサイクル管理](/docs/ja/sandboxes#sandbox-lifecycle) ポリシーを適用します。

### コンピュートプレーン \{#compute-plane\}

コンピュートプレーンは、サンドボックスが実行されるインフラストラクチャ層です。サンドボックスは [ランナー](#sandbox-runners) 上で実行され、ランナーは複数のサンドボックスを専用リソース付きでホストし、共有または専用の [リージョン](/docs/ja/regions) にわたって水平スケールします。コンピュートプレーンは次のコンポーネントで構成されます。

* [Sandbox runners](#sandbox-runners) はサンドボックスを専用リソース付きでホストします
* [Sandbox daemon](#sandbox-daemon) は各サンドボックス内でのコード実行および環境へのアクセスを提供します
* [Snapshot store](#snapshot-store) はサンドボックスのスナップショットイメージを保存します
* [Volumes](#volumes) はサンドボックス間で共有される永続ストレージを提供します

#### サンドボックスランナー \{#sandbox-runners\}

ランナーは、Daytona のコンピュートプレーンを支えるコンテナノードであり、サンドボックスのワークロードを実行するための基盤インフラストラクチャを提供します。各ランナーは、ジョブを取得するためにコントロールプレーンの API をポーリングし、サンドボックスの作成、起動、停止、破棄、リサイズ、バックアップといった各種操作を実行します。ランナーは、スナップショットとボリュームのデータについては S3 互換オブジェクトストレージと、スナップショットイメージについては内部 Docker レジストリと連携します。

各サンドボックスは、プロセス、ネットワーク、ファイルシステムマウント、およびプロセス間通信ごとに独立した Linux 名前空間を持つ分離されたコンテナとして実行されます。各ランナーは、サンドボックスごとに専用の vCPU、RAM、およびディスクリソースを割り当てます。

#### Sandbox daemon \{#sandbox-daemon\}

サンドボックスデーモンは、各サンドボックス内で動作するコード実行エージェントです。このデーモンは [Toolbox API](/docs/ja/tools/api#daytona-toolbox) を公開しており、ファイルシステムや Git 操作、プロセスおよびコードの実行、コンピューター使用、ログストリーミング、ターミナルセッションなど、サンドボックス環境への直接アクセスを提供します。

各サンドボックスデーモンは Docker Engine コンテナランタイムを管理し、サンドボックスが分離された自身の環境内でコンテナをビルドおよび実行できるようにします。デーモンはランナーのコンテナマネージャと通信し、サンドボックスの状態を報告し、コントロールプレーンからライフサイクルコマンドを受信します。

#### スナップショットストア \{#snapshot-store\}

スナップショットストアは、OCI Distribution 仕様に従ってサンドボックスのスナップショットイメージを保存する、内部の OCI 準拠レジストリです。ランナーは新しいサンドボックスを作成する際に、このストアからスナップショットイメージを取得します。ストアはバックエンドとして S3 互換のオブジェクトストレージを使用します。

#### ボリューム \{#volumes\}

[ボリューム](/docs/ja/volumes) は、サンドボックス間で共有可能な永続ストレージを提供します。各ボリュームは S3 互換のオブジェクトストレージをバックエンドとして使用し、読み書き可能なディレクトリとしてサンドボックスコンテナにマウントされます。複数のサンドボックスが同一のボリュームを同時にマウントできるため、サンドボックス間でデータを共有できると同時に、サンドボックスのライフサイクルとは独立してデータを永続化できます。

### コンテナレジストリ \{#container-registry\}

コンテナレジストリは、サンドボックスのベースイメージの取得元として機能します。[スナップショット](/docs/ja/snapshots)を作成する際、スナップショットビルダーは指定されたイメージを外部レジストリから取得し、ランナーが利用できるよう内部のスナップショットレジストリにプッシュします。Dockerfile ベースのスナップショットでは、`FROM` ディレクティブで参照される親イメージも、ビルド時に設定済みのソースレジストリから取得されます。Daytona は、任意の OCI 互換レジストリをサポートしています:

* [Docker Hub](/docs/ja/snapshots#docker-hub)
* [Google Artifact Registry](/docs/ja/snapshots#google-artifact-registry)
* [GitHub Container Registry (GHCR)](/docs/ja/snapshots#github-container-registry-ghcr)
* [Private registries](/docs/ja/snapshots#using-images-from-private-registries): OCI distribution specification を実装した任意のレジストリ
